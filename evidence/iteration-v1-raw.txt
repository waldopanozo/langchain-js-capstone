### PR Description

**Dev-142: Add Rate Limiting to Public Booking API**

**Priority**: high
**Parents can spam the booking endpoint during peak registration. We need per-IP rate limits on POST /api/bookings and a clear 429 response with Retry-After.**
 
This PR introduces a new middleware called `RateLimitMiddleware` that adds rate limiting to the public booking API endpoints. The goal is to ensure that:

1. **Maximum Requests Per IP:**
   - If a user exceeds the allowed number of requests, they will receive a 429 response with a Retry-After header indicating the time until the next limit.

2. **Configuration:**
   - Rate limiting can be configured via environment variables.
   - The maximum number of requests per IP (e.g., `MaxRequests` in this case) is configurable via an environment variable named `RATE_LIMIT_MAX_REQUESTS`.
   - Existing successful bookings are unaffected during rate limiting attempts.

3. **Unit Tests:**
   - Unit tests have been added to cover the following scenarios:
     - POST /api/bookings returns 429 when limit exceeded
     - The limit is configurable via environment variable
     - Existing successful bookings are unaffected

**Acceptance Criteria**: 
- POST /api/bookings returns 429 when limit exceeded
- Limit is configurable via environment variable
- Existing successful bookings are unaffected
- Unit tests cover limit boundary
